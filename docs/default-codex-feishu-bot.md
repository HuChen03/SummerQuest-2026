# 默认 Codex 飞书机器人与自定义说明

实验室为每位同学准备一套默认飞书机器人代码，使用 Codex 作为主要模型与工具调用执行器。同学可以直接使用这套默认实现，也可以在此基础上修改，或完全替换为自己的机器人方案。

本说明介绍默认实现的预期行为和资料约定，不是限制自定义机器人的协议，也不是 A6 作业内容。A6 的主题和要求会在后续单独公布。

## 默认实现

当前默认机器人提供：

- 飞书消息接收、回复和多轮对话。
- 基于 Codex 的回答生成。
- 读取配置指定的公开 `PROFILE.md`。
- 在个人事实后标注 `PROFILE` 章节来源。
- 资料不足时明确说明，不编造个人经历或作业结论。

仓库唯一保留的机器人代码位于 [`starter/default_codex_bot`](../starter/default_codex_bot/README.md)。它直接使用飞书 SDK 长连接，不依赖 `lark-cli` 或用户 OAuth；Codex 的模型与 AIGW 供应商参数由执行代码传入，不读取或修改用户的 `~/.codex/config.toml`。同一进程同时提供 `/ping <nonce>` 健康检查和默认 Codex 自动问答。

AIGW 地址不写入公开仓库。同学应从组织内飞书课程文档获取地址，并通过本地 `AIGW_BASE_URL` 环境变量注入；不得把地址复制到 GitHub、公开日志或机器人配置文件。

## 默认资料来源

当前基线只读取启动配置中明确指定的公开 `PROFILE.md`。计划扩展的资料边界包括：

1. `students/<同学真名>/` 中的公开 GitHub 内容。
2. `PROFILE.md` 登记的个人飞书主页。
3. 各作业 `assignments/<A编号>/README.md` 登记的组内飞书补充文档。

当前代码不会自动打开 `PROFILE.md` 或作业 README 中的飞书链接。后续增加组内飞书检索时，必须使用明确授权的服务身份，并继续遵守下述资料边界；飞书链接存在不代表机器人天然获得权限。

每位同学的机器人名称、App 信息、代码、模型、部署状态和消息验收结果都不需要写入 GitHub。需要个性化配置时，可直接在服务器、机器人代码或组内飞书文档中维护。

## 同学可以任意自定义

同学可以按自己的需要：

- 修改默认 prompt、身份描述和回答风格。
- 增删检索、搜索、代码执行或其他工具。
- 替换 Codex，使用其他模型、框架、编程语言或部署方式。
- 改写消息处理、记忆、多轮对话、引用和交互界面。
- 完全不使用默认代码，从零实现自己的飞书机器人。

自定义实现不需要保持与默认代码相同的内部结构。只要继续使用实验室资源或处理组内材料，就必须遵守仓库的公开性、凭据和权限规则。

## 不随自定义而改变的安全要求

- App Secret、Token、Cookie、密码和私钥不得提交到 GitHub 或飞书正文。
- 公开 GitHub 内容与组织内公开的飞书正文必须分开处理。
- 不得读取其他同学、未登记文档或未授权实验室资源。
- 不得把组内材料自动复制到公开回答、日志、索引或仓库。
- 自定义机器人产生的日志和缓存也不能记录凭据、完整私聊内容或内部地址。

这些要求来自实验室资料与账号安全边界，与使用 Codex、其他模型或完全自研实现无关。

## 与课程作业的关系

- A0：不要求创建、部署或验收飞书机器人；本实现仅作为自愿实践资料。
- A6：当前只做占位预告，具体主题与提交要求后续另行发布，不默认等同于机器人开发任务。
